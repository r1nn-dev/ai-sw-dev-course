Part 1 핵심

안쪽 while True 루프가 Tool 호출이 없을 때까지 반복하기 때문에,
사용자의 한 번의 입력으로 LLM이 여러 번 Tool을 호출할 수 있다.

Part 2: Tool 자리에 Agent가 들어갈 수 있다

단순 함수(Tool) vs 또 다른 에이전트(Agent-as-Tool) — 구조의 확장

단순 Tool (함수)

입력: {"filename": "main.py"}
↓
read_file() 함수 실행
↓
결과 반환 (파일 내용)

정해진 로직을 1번 실행하고 끝
판단 능력 없음

VS

Agent-as-Tool (에이전트)

입력: "이 코드의 버그를 찾아줘"
↓
Sub-Agent의 LLM 호출
↓
Sub-Agent가 자체 Tool 사용
↓
필요하면 반복... (자체 Agent Loop)
↓
최종 결과 반환

자체 LLM + Tool + Loop을 가짐
스스로 판단하고 행동

Part 3: Multi-Agent 실제 동작 흐름

메인 Agent가 "코드 리뷰 Agent"를 Tool처럼 호출하는 예시

사용자 입력
"main.py를 읽고, 코드 리뷰도 해줘"
Main Agent (Orchestrator)

1. LLM 호출

"먼저 파일을 읽어야겠다"
→ tool: read_file({"filename":"main.py"})

2. read_file 실행 (단순 Tool)

파일 내용을 반환

3. LLM 호출 (2회차)

"파일 내용을 받았으니 리뷰를 맡기자"
→ tool: code_review_agent({...})

4. code_review_agent 호출 (Agent Tool!)

이 Tool은 함수가 아니라 또 다른 Agent
자체 LLM + Tool + Loop을 가지고 동작 →

5. LLM 호출 (3회차)

Sub-Agent 결과를 받아서 최종 답변 생성

6. 최종 답변

"파일 내용은 ...이고, 리뷰 결과는 ...입니다"

펼쳐보면
Sub-Agent: code_review_agent 내부
Sub LLM 호출 1
"코드 구조를 먼저 파악하자"
→ tool: list_files({"path":"."})
list_files 실행
프로젝트 구조 파악
↺ 자체 루프 계속
Sub LLM 호출 2
"관련 파일을 더 읽어보자"
→ tool: read_file({"filename":"utils.py"})
read_file 실행
utils.py 내용 확인
↺ 자체 루프 계속
Sub LLM 호출 3
"충분히 분석했다. 리뷰 결과를 정리하자"
Sub-Agent 결과 반환
"버그 2건 발견, 리팩토링 제안 3건..."
→ Main Agent에게 tool_result로 전달

실제 예시: Tool Registry 확장

Tool 이름 유형 내부 동작 실제 사례
read_file 단순 함수 파일 읽고 반환 (1회) 현재 main.py의 read_file_tool()
list_files 단순 함수 디렉토리 조회 (1회) 현재 main.py의 list_files_tool()
code_review_agent Agent 자체 LLM+Tool로 다단계 리뷰 Claude Code의 Agent tool
research_agent Agent 웹 검색 + 요약을 반복 Perplexity, ChatGPT Deep Research
deploy_agent Agent 빌드 → 테스트 → 배포를 자동화 CI/CD 파이프라인 에이전트

핵심 통찰

Tool의 인터페이스는 동일하다: 입력 → 실행 → 결과 반환

단순 함수든, 내부에 자체 Agent Loop을 가진 에이전트든,
Main Agent 입장에서는 똑같은 Tool일 뿐이다.

이 "Tool = Agent가 될 수 있다"는 원리가
Multi-Agent 시스템의 기초이자,
복잡한 작업을 분업할 수 있게 만드는 핵심 구조다.
